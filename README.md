# Business Policy Compliance and Customer Resolution Environment

This repository is an OpenEnv-style evaluation environment for policy-aware support agents.  
Agents must route, prioritize, escalate, request clarification, flag fraud, and respond while following explicit business rules.

## What Makes This Environment Useful

- Multi-domain tickets across billing, technical support, returns, legal, customer success, and spam.
- 52 canonical scenario families generated by a `ScenarioFactory` (`14 easy`, `18 medium`, `20 hard`).
- Task-level resets produce seeded variants with randomized refund thresholds, SLA pressure, timestamp spacing, and policy versions so agents cannot rely on fixed-text memorization.
- Adaptive resets use recent performance to select the next difficulty tier when no explicit task or scenario is requested.
- Session-isolated API for concurrent judges and agents using `X-Session-Id`.
- Multi-turn state machine (`episode_phase`) for clarification-sensitive workflows.
- Policy versioning (`v1` and `v2`) to prevent fixed-policy memorization.
- Upcoming policy transitions are surfaced in the observation via `policy_transition_to` and `steps_until_policy_change`.
- Clarification snapshots unlock only after a substantive clarifying question clears the grading threshold.
- Partial observability on selected hard tickets hides some account flags while exposing the count via `hidden_flags`.
- `consult_specialist` restores a human-in-the-loop action with a limited per-episode budget and explicit reward cost.
- Optional LLM judging for `draft_response` can be enabled through environment variables, while the default benchmark path remains deterministic and offline.
- Browser-based Gradio judge UI.

## Policy Sets

Observations include `policy_version` and the active rule list.

`v1`:
- Refunds over `$500` require escalation.
- VIP customers require `high` or `urgent` priority.
- Issues open over `72` hours require `urgent` priority.
- Complaints mentioning legal action or lawsuits require immediate escalation.
- Suspended accounts must be routed to the billing team.

`v2` (all `v1` rules plus):
- Premier accounts require same-day response regardless of issue type.
- Fraud indicators require `flag_fraud` before any resolution action.

## Observation Space

Each observation includes:
- Current email and visible thread.
- Sender tier, account flags, refund amount, and issue age.
- `hidden_flags`, which counts policy-relevant account flags intentionally withheld from the agent in some hard scenarios.
- `emails_remaining`, which counts hidden future customer messages not yet revealed to the agent.
- `policy_transition_to` and `steps_until_policy_change`, which expose upcoming policy drift when a scenario changes rule sets mid-episode.
- `specialist_consult_budget_remaining`, `specialist_consults_used`, and `specialist_notes` for human-in-the-loop consultation.
- Action history, policy rules, and task objective.
- `policy_version` and `episode_phase`.

## Action Space

- `categorize`
- `set_priority`
- `draft_response`
- `escalate`
- `mark_spam`
- `request_info`
- `flag_fraud`
- `snooze`
- `consult_specialist`

`flag_fraud` requires `fraud_reason`.  
`snooze` requires `snooze_hours` and can trigger SLA penalties if it crosses the 72-hour threshold.
`consult_specialist` requires `specialist_team` and spends limited episode budget.

## Reward Table

| Component | Trigger | Value | Notes |
| --- | --- | --- | --- |
| Valid action | Any schema-valid action | +0.05 | Always granted for well-formed actions |
| Policy penalty | Action contradicts active rule | -0.2 | Immediate |
| SLA crossed during snooze | Snooze pushes age past 72h | -0.1 | Immediate |
| Specialist consult | `consult_specialist` action | -0.05 | Immediate cost for human-in-the-loop support |
| Fraud missed | No `flag_fraud` in fraud scenario | -0.15 | Episode-end penalty |
| Partial progress | Correct work completed so far | 0.0-current task score | Returned each step |
| Final grader score | Episode complete | 0.0-1.0 | Full grader runs on `done` |
| Efficiency bonus | Finished in <= half allowed steps | +0.1 | Episode-end bonus |
| Redundancy penalty | Repeated action types | -0.05 per repeat | Episode-end penalty |
| Invalid action | Schema-invalid action payload | -0.1 | Immediate step penalty |

Intermediate rewards may be negative. Final episode rewards are clamped to `[0.0, 1.0]`.  
Step details are available in `info["reward_breakdown"]`.

## API

FastAPI routes:
- `GET /health`
- `GET /tasks`
- `POST /reset`
- `POST /step`
- `GET /state`
- `DELETE /session`

Session header:
- `X-Session-Id` (optional, defaults to `default`)

Important behavior:
- `/step` requires an existing session (call `/reset` first for that session).
- `/state` is session-scoped.
- Public `/state` is sanitized and does not expose grading ground truth.

## Gradio Judge UI

`gradio_app.py` starts:
- FastAPI server on `7860`
- Gradio UI on `7861`

The UI supports scenario selection, reset, all action types, and live reward/component readouts.

## Local Run

Install:
```bash
python -m venv .venv
./.venv/bin/pip install -e ".[dev]"
```

Optional OpenAI baseline support:
```bash
./.venv/bin/pip install -e ".[dev,openai]"
```

Optional Anthropic-backed LLM baseline:
```bash
./.venv/bin/pip install -e ".[dev,llm]"
ANTHROPIC_API_KEY=your-key ./.venv/bin/python baseline.py --agent llm --model claude-3-5-sonnet-latest
```

Optional hard-task response judge:
```bash
JUDGE_API_KEY=your-key BUSINESS_POLICY_ENV_USE_LLM_JUDGE=true ./.venv/bin/python -m pytest tests/ -v
```

Run API only:
```bash
./.venv/bin/uvicorn business_policy_env.server:app --host 0.0.0.0 --port 7860
```

Run Gradio + API:
```bash
./.venv/bin/python gradio_app.py
```

Run checks:
```bash
./.venv/bin/ruff check .
./.venv/bin/mypy business_policy_env/
./.venv/bin/python -m pytest tests/ -v
./.venv/bin/openenv validate
```
